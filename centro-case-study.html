Background

The project that has outlasted three product names and two org restructures.

CENTRO Engineer is Weatherford's real-time drilling engineering platform — a complex application used by drilling engineers and mud engineers in the field to monitor wellbore conditions, run hydraulics and mechanics analyses, plan wells, and manage data acquisition in real time. It is not software you pick up in an afternoon.

I was brought onto the project — then called MC2 — about ten years ago to work alongside an existing technical writer. It became clear within weeks that the existing documentation wasn't salvageable. I rewrote everything from scratch. I've owned the project ever since, through a full rename to OneSync, a split into separate products with separate dev teams, another rename to CENTRO Engineer and Victus, and a significant expansion of the product portfolio.

Through all of it, I was the sole source of documentation continuity.

The Problem

Documentation written for the wrong audience, using the wrong tools, in the wrong way.

The Flare project I inherited was built on a template I had originally created for another project. The previous writer had it — but clearly didn't understand how to use Flare. There was no content reuse: no snippets, no variables, no conditions. File naming was inconsistent. The project was littered with unused topics and orphaned images. The architecture was, in a word, chaotic.

The content itself had a more fundamental problem: it misunderstood who it was for. Topics that should have explained how to use the software instead explained basic industry concepts to specialists who had spent careers mastering those concepts. One topic — for a module used by mud engineers to configure drilling fluids — contained a lengthy explanation of what a mud balance is and how it works. Written for mud engineers. Other topics contained nothing but a screenshot.

The core failure

The previous documentation didn't distinguish between what users needed to know and what the writer had just learned. That confusion — between domain education and software documentation — produces content that's useless to experts and still doesn't explain the software. It's the most common mistake an inexperienced technical writer makes when dropped into a specialized industry.

It took two years to rewrite every topic — while simultaneously keeping pace with active development on a product that was shipping new features throughout. At the end of it, not a single word from the original documentation remained.

Scope & Scale

One Flare project. Five applications. Multiple output formats.

At its most complex, the single Flare project I managed produced separate user guides for five distinct applications: MC2 Monitoring, MC2 Operation, MC2 Simulation, MC2 Planning, and MC2 Player — plus MC2 Management and, later, MC2 TPS. Each application had its own audience, its own workflows, and its own documentation requirements. All of it lived in one project, managed by one writer, using conditions and content reuse to keep outputs accurate and maintainable without duplicating effort.

As the product evolved, applications were renamed, combined, retired, and eventually split into separate products with separate codebases and separate Flare projects. I managed that transition — restructuring the content architecture as the products diverged — without losing documentation continuity for any of them.

Featured Work

Context-sensitive help: both sides of the implementation.

Most technical writers who "implement CSH" handle the Flare side: setting up the map file, assigning aliases, configuring the output. The other half — placing the triggers in the application code — typically falls to a developer.

I did both.

The application is built in C#. When we implemented context-sensitive help, I wasn't just responsible for the Flare configuration. I was also responsible for identifying where in the codebase each trigger needed to live and placing it there — in a codebase I was still learning at the time. That meant reading the application code well enough to understand which UI elements mapped to which help topics, and wiring them together correctly across every screen in the application.

The system launched with approximately 300 triggers. It now has around 500. Every new screen that ships requires a new trigger, implemented on both sides — still by me.

Video tutorials: a complete production program built from a one-sentence directive.

The ask was simple: "We want videos." That was it. No scope, no format, no process, no equipment list, no guidance on content.

What followed was a program I built entirely from scratch: researching and selecting production software (Camtasia) and screen capture tools (SnagIt), determining what equipment was needed, establishing a production workflow, deciding which workflows were worth covering, scripting and recording each video, and then integrating the finished videos into the help system — including the CSS work required to embed and display them correctly within the Flare output.

The result was a library of 17 tutorial videos focused on setup and configuration workflows: the foundational tasks users need to complete before the software can do what it's designed to do. The kind of content that prevents support tickets from being filed in the first place.

Professional Recognition

Speaking at the field's premier conference. Twice.

The work on CENTRO Engineer wasn't just professionally formative — it generated ideas worth sharing with the broader technical communication community. I presented at the STC Technical Communication Summit in both 2018 and 2019, the largest annual conference for technical communicators in the world.

The 2018 talk grew directly out of the documentation architecture challenges I'd solved on this project: managing a single Flare project producing five separate application outputs, under active development, with strict naming conventions holding the whole system together. The 2019 talk extended that thinking into a broader framework for taxonomy design in technical communication.
