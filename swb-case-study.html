Background

Two documentation problems at once, in a domain I'd never touched.

Surveyor's Workbench (SWB) is Weatherford's survey correction platform, used by Real-Time Operations Center (RTOC) engineers to apply Multi-Station Analysis (MSA) and sag corrections to raw MWD survey data — ensuring that borehole position and orientation measurements are as accurate as possible during active drilling operations. Getting these corrections right matters: they feed directly into well positioning, anti-collision calculations, and steering decisions.

When I came onto the project, the situation was this: SWB Desktop had been in use for years, supported by a 2015 field instruction document that covered some procedures and conceptual background but had no field definitions, hadn't been updated in nearly a decade, and wasn't a user guide in any meaningful sense. At the same time, a new web-based application — SWB Orchestrator — had just been developed as the first phase of a modernization effort, and had no documentation whatsoever.

Both problems needed solving. Simultaneously. In a technical domain I was starting from scratch in.

The Domain Challenge

Learning a completely new class of survey data — from the physics up.

Surveys had been part of my work on CENTRO Engineer, but in a limited way — well setup data expressed as measured depth, inclination, and azimuth. Three values. Straightforward.

SWB operates on a fundamentally different class of survey data. Instead of processed inclination and azimuth values, it works with raw six-axis sensor readings: three magnetic field components (Bx, By, Bz) and three gravitational components (Gx, Gy, Gz). These are used to identify and correct for magnetic interference from the drill string, sensor misalignment, and the physical deflection of the BHA under gravity — the sag effect — all of which introduce error into borehole position calculations if left uncorrected.

What domain learning actually looks like

Understanding MSA correction well enough to document it meant learning why it's necessary (magnetic interference from the BHA corrupts raw survey data), how it works (iterative statistical optimization across multiple survey stations), what can go wrong (one bad survey can corrupt the entire solution), and what the outputs mean (corrected azimuth and inclination values with quantified uncertainty). None of this came from a manual. It came from developer conversations, SME sessions, the 2015 field instruction, and a lot of time with the software itself.

This is the part of technical writing that doesn't show up on a resume: the months of learning that happen before a single topic gets written. By the time the SWB documentation was complete, I understood survey correction well enough to explain it to an RTOC engineer who'd never used the software — which is exactly what the documentation needed to do.

Information Architecture

Five roles. Four guides. One Flare project.

SWB Orchestrator presents a different user experience depending on the role assigned to your account. The five roles — Super User, RTOC Admin, RTOC User, Rig Engineer, and Rig User — each see a different interface with different capabilities. A single monolithic user guide would have been nearly useless: every user would spend most of their time reading content that didn't apply to them.

The solution was role-based user guides: separate help outputs for each role, where clicking Help opened the guide specific to your account type. In practice, this meant four guides — Super User and RTOC Admin have limited scope and lower user counts, so their documentation is appropriately scoped — with the primary depth invested in the RTOC User and Rig Engineer guides, which cover the core workflows the platform is built around.

The Rig User role presented a content reuse opportunity: Rig Users have identical access to Rig Engineers, but everything is read-only. Rather than maintaining a separate guide, I built the Rig Engineer guide and used MadCap Flare conditions to filter out any content that didn't apply to read-only users. One source, two outputs, no duplication.

Featured Work

SWB Orchestrator: greenfield documentation for a brand-new application.

SWB Orchestrator had no prior documentation — no field instruction, no internal wiki, nothing. Every topic was written from scratch, starting with a fundamental question: what does this application actually do, and in what order do users do it?

The answer required working through the complete workflow with developers and SMEs, then validating it hands-on. The result was a structured set of role-based guides covering the full RTOC User and Rig Engineer workflows: creating jobs and runs, uploading raw survey data, connecting runs to the desktop application, processing MSA and sag corrections, sending corrections back to the web, and managing the run lifecycle.

Determining the correct workflow was itself a documentation challenge. The integration between the web application and SWB Desktop is functional but not frictionless — there are sequencing dependencies between what the rig does, what the RTOC does, and what each application's status reflects at any given moment. Getting that sequence documented accurately required understanding it at a level beyond what any single SME had articulated.

SWB Desktop: from a 2015 field instruction to a complete user guide.

The SWB Desktop guide had a different starting point than the Orchestrator guides: a 64-page field instruction document written in 2015 that covered the MSA and sag correction workflow in reasonable procedural depth, but contained no field definitions, hadn't been updated in nearly a decade, and wasn't structured for software documentation.

The field instruction was useful raw material — it provided solid conceptual background on MSA and sag correction, and its procedural content reflected how experienced RTOC engineers actually worked. But it needed to be transformed: restructured into topic-based help, supplemented with the field definitions and UI documentation that were entirely absent, updated for a decade of software changes, and extended to cover the new web integration workflows that didn't exist when the original document was written.

The web integration section — covering how RTOC users connect SWB Desktop to SWB Orchestrator, process corrections, and send results back to the web — was written entirely from scratch. It documents a workflow that was brand new to users who had operated the desktop application in isolation for years, and required enough understanding of both applications to explain how they interact.

Interactive demos: ~25 Tourial walkthroughs, end to end.

In addition to the written documentation, I produced approximately 25 interactive Tourial demos covering the major workflows across both SWB Orchestrator and SWB Desktop. I determined which workflows warranted demo coverage, produced each demo, and integrated them into the help system — with the SWB Orchestrator workflows prioritized first, given that the web application was entirely new to users.

The demos were built within a provided template that established the visual style and intro/outro screens. Everything else — content selection, workflow sequencing, callout text, integration into the help system — was mine. All demos are embedded directly in the user guide, accessible inline alongside the relevant written procedures.

A Strategic Decision

One Flare project for both products — by design.

SWB Desktop and SWB Orchestrator are separate applications, and I could have maintained them in separate Flare projects. I chose not to. The SWB modernization effort will eventually bring most or all of the Desktop functionality into the web application — which means the two products will converge over time, and so will their documentation.

Keeping both in a single Flare project makes that transition easier: shared content can be reused without cross-project complexity, and updates that affect both products can be maintained in one place. It also made it straightforward to add the web integration section to the Desktop guide — a section that bridges the two products — without duplicating content that already existed in the Orchestrator guides.

It's the kind of decision that doesn't save time today but pays dividends over the life of the project.
